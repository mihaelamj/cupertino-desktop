🧠 Cupertino Data State Language
CDSL (State)
Operates **BEFORE the UI**. Drives the view models programmatically to test data loading, search debouncing, sorting, and state transitions without rendering widgets.

📐 Grammar EBNF & Computed LL(1) Lookaheads

CDSL EBNF Grammar

P  ::= S P | ε
S  ::= D | A | W
D  ::= 'dispatch' IDENT '(' V ')'
A  ::= 'assert' 'vm' IDENT Op V
W  ::= 'await' 'tasks'
V  ::= STRING | NUMBER | BOOLEAN | 'nil' | L | IDENT
L  ::= '[' E ']'
E  ::= V E' | ε
E' ::= ',' V E' | ε
Op ::= '==' | '!=' | 'contains'

CLL EBNF Grammar

P  ::= S P | ε
S  ::= D | A
D  ::= 'device' IDENT 'in' IDENT 'with' IDENT
A  ::= 'assert' 'ui' IDENT Op V
V  ::= STRING | NUMBER | BOOLEAN | 'nil' | IDENT
Op ::= '==' | '!=' | 'shows' | 'hides'

Lookahead FIRST+ Sets

Production Rule Lookahead Set (FIRST+) Parse Decision Action
Statement → DeviceDecl { 'device' } Instantiates device profile, size classes, and layouts
Statement → ActionStmt { 'dispatch' } Dispatches programmatic view-model actions and keystrokes
Statement → AssertStmt { 'assert' } Validates data-layer or visual-hierarchy properties
Statement → WaitStmt { 'await' } Awaits asynchronous tasks (coalescing, debouncing, loading)
Operator → hides / shows { 'hides', 'shows' } Performs visibility/existential validation checks in mimic layer

🛡️ Matt Polzin / OpenAPIKit Validation Principles

Both languages process validations using Matt Polzin's OpenAPIKit composable idioms. Our validation engine implements these 11 constraints:

Principle Design Rule Description Swift Implementation Details
1. Parse First, Validate Second Parser yields a structured, strongly typed AST. Compiler runs validation as a distinct second pass. Validator<Subject>.run(ast)
2. Composable Rules Rules are declared as static properties conforming to a validation protocol. static let assertedPropertiesExist: Validation<Subject, Doc>
3. Derivation of Errors Error messages are derived automatically by prefixing "Failed to satisfy: " to the positive description. ValidationError(description: "Failed to satisfy: " + rule.description)
4. Contextual Coding Paths Errors trace their path down the AST using a list of coding keys. codingPath: [CodingKey] (traces index, fields, nested blocks)
5. Erased Type Dispatch Heterogeneous rule lists are held in type-erased wrappers that safely cast target structures. AnyValidation<Doc> wrapper filters out non-matching types

🔄 Headless Testing Lifecycles & Swift vs. CLIL Comparisons

Detailed breakdown of testing lifecycles showing side-by-side CLIL scripts and their corresponding Swift Unit Test assertions.

Phase 1: App Boot & Databases List Loading

Behavior: User launches the app. A list of available database registries is shown. VM connects concurrently; visual layout starts synchronously.

CLIL Script
device iPhone in portrait with compact
assert ui activeView == "Databases"
assert ui navigationStackDepth == 0
Swift API Test
let vm = FrameworkBrowser.ViewModel()
#expect(vm.selectedSource == nil)
#expect(vm.connectionState == .connecting)

Phase 2: Database Selection & Framework List Loading

Behavior: User selects a database source. VM triggers a background SQLite query to read available frameworks.

CLIL Script
dispatch selectSource("apple-docs")
assert vm activeSource == "appleDocs"
assert ui activeView == "Frameworks"
assert ui navigationStackDepth == 1
Swift API Test
vm.selectSource(.appleDocs)
#expect(vm.selectedSource == .appleDocs)

let simulator = CLILSimulator(frameworks: vm)
#expect(simulator.ui.navigationStackDepth == 1)

Phase 3: Framework Selection & Document List Loading

Behavior: User selects a framework (e.g. SwiftUI). The VM loads the document array from the FTS5 index.

CLIL Script
dispatch selectFramework("swiftui")
assert vm selectedFrameworkID == "swiftui"
assert ui activeView == "Documents"
assert ui navigationStackDepth == 2
Swift API Test
vm.selectFramework("swiftui")
#expect(vm.selectedFramework?.id == "swiftui")

Phase 4: Document Selection & Reader Rendering

Behavior: User selects a document. The reader pane parses GFM raw markdown on a background thread and updates view state.

CLIL Script
dispatch selectDocument("apple-docs://swiftui/view")
assert vm documentState == "loading"
await tasks
assert vm documentState == "loaded"
assert ui activeView == "Reader"
assert ui navigationStackDepth == 3
Swift API Test
vm.selectDocument(Model.DocURI("apple-docs://swiftui/view")!)
#expect(vm.documentState == .loading)
try await Task.sleep(nanoseconds: 50_000_000)
#expect(vm.documentState == .loaded)

Phase 5: Search Query, Debouncing, & FTS5 Candidates

Behavior: User searches text. Keystroke entries update search text synchronously, while debounced FTS5 query executes after 250ms asynchronously on a background queue.

CLIL Script
dispatch search("S")
dispatch search("Sw")
dispatch search("Swi")
assert vm isLoading == false
await tasks
assert vm state == "loaded"
assert vm results contains "SwiftUI"
Swift API Test
searchVM.text = "S"
searchVM.runDebounced()
#expect(searchVM.isLoading == false)
try await Task.sleep(nanoseconds: 300_000_000)
#expect(searchVM.results.contains(where: { $0.title == "SwiftUI" }))
